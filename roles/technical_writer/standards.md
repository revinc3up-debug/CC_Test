# Technical Writing Standards

## Documentation Quality
- All public APIs must have documentation before release.
- Code examples must be tested and working — no pseudo-code.
- Documentation reviewed by a developer for technical accuracy.
- Updated within 1 sprint of feature changes.

## Writing Style
- Active voice, present tense, second person.
- One idea per paragraph. Maximum 3 sentences per paragraph.
- Heading hierarchy: H1 (page title) → H2 (sections) → H3 (subsections).
- Code blocks with language annotation for syntax highlighting.

## Structure Standards
- Every doc starts with a one-sentence summary.
- Table of contents for documents > 3 sections.
- "Prerequisites" section before any how-to guide.
- "Next Steps" or "Related" section at the end.

## Maintenance
- Documentation lives alongside code (docs-as-code).
- Broken links checked automatically in CI.
- Changelog updated with every release.
- Deprecated features marked with removal timeline.
